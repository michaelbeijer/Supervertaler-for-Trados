# Memory Bank Management – Feature Spec

**Status:** draft, 2026-04-15
**Target phase:** Step 5i phase B (follow-up to the multi-bank work that shipped in v4.19.0)
**Scope:** Trados plugin. Python Workbench port should follow the same contract (same on-disk semantics, same collision policy).

## Motivation

Multi-bank support shipped in v4.19.0 with toolbar switching and in-plugin bank creation, but three operations still require closing Trados and using File Explorer:

- **Rename** a bank
- **Delete** a bank
- **Merge** one bank into another
- **Move** selected articles from one bank to another

The first real merge exercise (MB, 2026-04-15 – consolidating four banks into `default`) exposed several non-obvious concerns that should shape the UI:

1. A bank can be a symlink or a git repo – destructive actions must detect and preserve that.
2. Collisions are not rare when users have organised work by client for a while and then decide to consolidate. In that one exercise, 3 of 95 files collided; in 2 of the 3 cases the *source* was richer than the target – an unconditional "overwrite" or "keep target" default would have silently lost data either way.
3. Some files in a bank are shipped scaffolding (`_EXAMPLE_*`, `06_TEMPLATES/`) and must never be copied between banks – they are identical by construction and would inflate the archive with noise.
4. `05_INDICES/*.md` are auto-generated by Distill and must be regenerated after a merge, not copied.
5. The `00_INBOX/_archive/` tree is provenance material (PDFs, xlsx, tmx, raw md) that needs to be preserved with clear source attribution, not flattened.

These constraints are not specific to one merge – they describe the steady state of how banks relate to each other.

## Non-goals

- **No bank-level "sync" or "replication"** between machines. That is a separate feature (cloud sync / Dropbox strategy) and out of scope here.
- **No git operations.** If the target bank happens to be a git repo, the plugin leaves working-tree changes staged for the user to commit themselves. We do not run `git add` or `git commit` on the user's behalf.
- **No cross-product memory-bank format migration.** Banks created by the Python Workbench and by this plugin already share the same on-disk layout; keep it that way.

## User-facing surface

### Entry point

Add a **"Manage banks…"** entry to the SuperMemory toolbar's existing bank dropdown, positioned directly above the `"+ New memory bank…"` sentinel. Selecting it reverts the combo to the previously active bank (same `_lastRealSelection` pattern) and opens a modal `MemoryBankManagerDialog`.

```

│                                                            │
│  ┌──────────────────────────────┬─────────┬────────────┐  │
│  │ Name                          │ Articles │ Size      │  │
│  ├──────────────────────────────┼─────────┼────────────┤  │

│  └──────────────────────────────┴─────────┴────────────┘  │
```

- The active bank is marked and cannot be deleted or renamed without first switching to another bank (the dialog offers to do that for you).
- Symlinked banks and git-tracked banks get a small indicator icon in the Name column with a tooltip explaining the implication.
- Article count excludes `_EXAMPLE_*`, `06_TEMPLATES/`, and `05_INDICES/` so it reflects real user content.

### Four operations

Each operation has the same shape: select target(s), preview the action in a dry-run summary dialog, confirm, apply, report.

## Operation: Rename

**Input:** existing bank name + new name
**Preview:** "Rename `<old>` → `<sanitised-new>`. Any references to `<old>` in other settings will be updated."
**Action:**
1. Sanitise new name via `UserDataPath.SanitizeBankName` (mirrors Python `sanitise_bank_name` – see `memory_bank.py` in the retired `supervertaler-assistant` repo for the canonical regex).
2. Refuse if target name already exists or sanitises to empty.
3. `Directory.Move` the folder.
4. If the renamed bank was active, update `AiSettings.ActiveMemoryBankName` and invalidate the cached `MemoryBankReader`.
5. Reload the dropdown.

**Edge cases:**
- If the bank folder is a **symlink**, move the link itself (Windows reparse point), do not chase the target. Log that the underlying folder at `<target>` was not moved.
- If the bank folder is a **git worktree**, refuse with a message pointing at `git mv` – renaming a tracked directory via filesystem breaks git's index.

## Operation: Delete

**Input:** bank name + confirmation
**Preview:** "Delete `<name>` (N articles, X MB). This cannot be undone from inside the plugin. A timestamped zip backup will be created at `memory-banks/_backups/<name>-<timestamp>.zip` first."
**Action:**
1. Refuse if the bank is active (offer to switch first).
2. Refuse if it is the only remaining bank (avoid leaving the user with no banks; the plugin's `EnsureDefaultBank` would recreate an empty one, but the user data is gone).
3. If the folder is a git repo or symlink, show a stronger warning – deletion will remove the git working tree or the link target respectively.
4. Create the backup zip.
5. `Directory.Delete(recursive: true)`.
6. Reload the dropdown.

**Backup location:** `memory-banks/_backups/` is a new convention. Add `_backups/` to the gitignore template shipped with the SuperMemory skeleton repo. Mention it in the SuperMemory `README.md`.

## Operation: Merge into…

This is the heart of the feature. The 2026-04-15 exercise is the reference.

**Input:** source bank + target bank + collision policy
**Collision policy choices:**
- **Archive source version** (recommended default) – target wins; source-bank copy lands under `<target>/00_INBOX/_archive/from-<source>/skipped_collisions/<folder>/<filename>` for manual review.
- **Overwrite with source** – source wins; target's version is archived symmetrically under `00_INBOX/_archive/pre-merge-<timestamp>/<folder>/<filename>`.
- **Rename with suffix** – both coexist; source file lands as `<filename> (from <source>).md`. Useful when both versions are valid but for different contexts.
- **Ask per collision** – modal prompt per conflicting file with a side-by-side line-count + first-N-lines diff preview and the three options above.

**What gets copied:**
- Everything in `00_INBOX/` (loose files and the full `_archive/` tree, the latter placed under `from-<source>/original_archive/`).
- All `.md` files in `01_CLIENTS/`, `02_TERMINOLOGY/`, `03_DOMAINS/`, `04_STYLE/` except `_EXAMPLE_*`.
- Any `.edtz` sidecar that accompanies a copied `.md` (same basename + `.edtz`). This is Supervertaler/Workbench edit metadata; it is local-only (gitignored) and belongs with the article it describes.

**What does not get copied:**
- `06_TEMPLATES/` – the target's templates are authoritative; source bank's are copies of a shipped set.
- `05_INDICES/*.md` – regenerate after the merge by triggering a Distill pass on the target.
- `.obsidian/` – each bank has its own Obsidian workspace state; merging these produces garbage.
- `_EXAMPLE_*` files – scaffolding; target already has them.

**Dry-run summary dialog:**
```
Merge depoortere → default

Will copy:     39 files     (to 01_CLIENTS, 02_TERMINOLOGY, 03_DOMAINS, 04_STYLE)
Archive:       19 files     (00_INBOX/_archive from depoortere → default)
Collisions:     3 files     [Review…]
Will regenerate: 3 index files (05_INDICES) – triggered by Distill after merge

Source bank (depoortere) will remain on disk. You can delete it later via Manage banks.

                              [Cancel]  [Merge]
```

**Clicking [Review…] on collisions** opens a list; each entry shows source path, target path, byte counts, first-line-of-diff, and the three per-file options. Bulk-apply "archive source" / "overwrite target" shortcuts at the top.

**After a successful merge:**
1. Refresh the target bank's cached `MemoryBankReader` index.
2. Trigger a background Distill pass on the target to regenerate `05_INDICES/*.md`.
3. Drop a confirmation banner into the AI Assistant chat with the summary counts.
4. Leave the source bank intact (Delete is a separate explicit action).

## Operation: Move articles…

For the finer-grained case where a user wants to pull specific articles from one bank to another without doing a full merge – e.g., one client's terminology was accidentally added to the wrong bank.

**Input:** source bank + tree selection of articles + target bank + collision policy (same three options as merge)
**UI:** a two-pane picker. Left pane = source bank folder tree with checkboxes per article. Right pane = target bank, read-only, for orientation. Bottom bar shows "12 articles selected – move to default".

**Action:**
- For each selected article: move (not copy) the `.md` file (and its `.edtz` sidecar if present) into the same relative path under the target bank. Collision policy applies.
- Update any `[[backlinks]]` inside the moved articles that reference other articles that were also moved – their relative paths do not change, but references to articles that stayed in the source bank become dangling. Surface those as a post-move warning listing the broken links, do not auto-fix.

**No auto-fix of `[[backlinks]]`** is deliberate: in Obsidian, dangling links are visually obvious and the user can decide whether to duplicate the referenced article into the target bank, re-point the link, or accept the break. Silent link rewriting would hide user intent.

## Implementation sketch

### Core (in `Settings/UserDataPath.cs`)

```csharp
public enum BankCollisionPolicy
{
    ArchiveSource,    // target wins; source archived
    OverwriteTarget,  // source wins; target archived
    RenameSource,     // both coexist, source gets " (from <bank>)" suffix
    AskPerFile        // caller decides per collision
}

public sealed class BankMergePlan
{
    public string Source { get; }
    public string Target { get; }
    public IReadOnlyList<string> FilesToCopy { get; }
    public IReadOnlyList<string> ArchiveToCopy { get; }
    public IReadOnlyList<BankCollision> Collisions { get; }
}

public sealed class BankCollision
{
    public string RelativePath { get; }      // e.g. "02_TERMINOLOGY/zwad → swath.md"
    public long SourceBytes { get; }
    public long TargetBytes { get; }
    public string FirstDiffLine { get; }     // precomputed preview
}

public static BankMergePlan PlanMerge(string sourceBank, string targetBank);
public static bool TryApplyMerge(BankMergePlan plan, BankCollisionPolicy defaultPolicy,
                                 IDictionary<string, BankCollisionPolicy> perFileOverrides,
                                 IProgress<string> progress, out string error);

public static bool TryRenameBank(string oldName, string newName, out string error);
public static bool TryDeleteBank(string name, out string backupZipPath, out string error);
public static bool TryMoveArticles(string sourceBank, string targetBank,
                                   IEnumerable<string> relativePaths,
                                   BankCollisionPolicy policy,
                                   out IReadOnlyList<string> brokenBacklinkWarnings,
                                   out string error);
```

`PlanMerge` is pure – it does no I/O writes – so the dry-run dialog can present its result before `TryApplyMerge` runs.

### UI

New WinForms dialog `Controls/MemoryBankManagerDialog.cs`. Pattern after the existing `TermLensSettingsForm` (same visual weight). Collision review pane reuses the side-by-side diff approach from Batch Translate's report viewer where practical.

### Wiring

- Toolbar dropdown gets a new sentinel item `"⚙ Manage banks…"`. Selection handler in `AiAssistantViewPart.OnManageBanksRequested` shows the dialog and, on close, fires `MemoryBanksChanged` so every bank-aware control refreshes.
- `Core/HelpSystem.cs`: map the new dialog to a new GitBook page `ai-assistant/super-memory/managing-banks.md`.

## Safety rails

1. **Always backup before destructive ops.** Delete makes a zip; merge with `OverwriteTarget` writes the target's conflicting files into `00_INBOX/_archive/pre-merge-<timestamp>/` on the target before overwriting.
2. **Refuse mid-operation if the bank is locked.** Trados could be mid-batch-translate against the source bank. Check for active Distill / Process Inbox / Batch Translate jobs referencing either bank and refuse until they finish. Progress indicator on those ops already exists; reuse the lock registry.
3. **Confirm via typed bank name on Delete.** Classic "type `<bankname>` to confirm" pattern. Rename and Merge do not need this – they are reversible (rename can be rename-back, merge leaves the source intact).
4. **Log everything.** Write a merge/move audit log at `<bank>/00_INBOX/_archive/_audit.log` recording what was copied/archived/overwritten and when. Simple tab-separated, append-only. Useful for "where did this file come from" investigations months later.

## Open questions

1. **Merge by default on first run for users upgrading from single-bank?** The `TryMigrateLegacySingleBank` path already handles pre-multi-bank installs. If a user has created multiple banks and then wants to consolidate, they have to do it manually today (or wait for this feature). Should the Manage banks dialog get a "consolidation wizard" mode that walks through merging N banks into one? Probably yes, but V2.
2. **How should the Python Workbench port share code?** The memory-bank helpers in `memory_bank.py` are the canonical reference implementation of sanitisation and skeleton folders. The C# port duplicates them in `UserDataPath.cs`. Merge/move/rename/delete logic should stay parallel between the two – worth a shared spec doc (this one) rather than trying to share code across runtimes.
3. **Undo window?** Could we keep, say, the last 3 merge operations reversible by preserving a manifest of what was copied/overwritten, and offering "Undo last merge" in the dialog? Adds complexity but materially changes user willingness to experiment. Put it on the list; ship without it first.
4. **Copying `.edtz` sidecars across banks.** These reference source material by path; a moved `.edtz` that points at a file in the old bank's `_archive/` would point at a dangling path. Needs a pass that either rewrites the path references inside `.edtz` or surfaces the dangling ones. Spec for the `.edtz` format lives in the Workbench repo – audit before shipping.
